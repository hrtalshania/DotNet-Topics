Blazor Server
vs WebAssembly

A complete, in-depth technical reference — architecture, network behaviour, DevTools visuals, use cases, and decision guide.

🔵 Blazor Server → SignalR / WebSocket 🟡 Blazor WASM → HTTP API calls ⚡ .NET 8+ Render Modes
01 — Core Concept

What are They, Exactly?

BLAZOR SERVER
  • Runs where? Entirely on the ASP.NET Core server
  • Client gets? Minimal HTML + blazor.server.js only
  • Updates sent via? SignalR WebSocket connection
  • DB access? Direct — no API layer needed
  • Download size? Very small initial payload (~50KB JS)
  • C# on client? Never — all runs on server
  • Needs a server? Yes — always, always
VS
BLAZOR WEBASSEMBLY
  • Runs where? Entirely in the browser (WebAssembly runtime)
  • Client gets? Full .NET runtime + all DLLs downloaded
  • Updates sent via? HTTP calls to a Web API
  • DB access? Never direct — must go through API
  • Download size? Large initial payload (5–20MB+)
  • C# on client? Yes — compiled to WASM bytecode
  • Needs a server? Only for data/APIs — UI works offline
02 — How It Works Internally

Architecture & Request Flow

🔵 Blazor Server — Step-by-Step Flow
1
Browser makes initial HTTP request Server responds with minimal HTML + blazor.server.js (tiny JS file)
2
SignalR connection is established (WebSocket) blazor.server.js initialises and opens a persistent WebSocket tunnel to /_blazor?id=xxx
3
Server builds a Render Tree in memory A full virtual DOM representation of the page is held server-side per user ("circuit")
4
User clicks a button / interacts The event is sent from browser → server as a tiny binary message over the WebSocket
5
Server processes event, updates Render Tree It compares Old Tree vs New Tree and computes only the DOM diff (delta changes)
6
Server sends binary diff back to browser Only the changed parts of the DOM are sent — not a full page reload. Browser applies the patch.
🟡 Blazor WebAssembly — Step-by-Step Flow
1
Browser makes initial HTTP request Server serves index.html, blazor.webassembly.js, and triggers download of the app
2
Browser downloads the full .NET runtime + DLLs dotnet.wasm (compiled .NET runtime), all app assemblies (.dll files), fonts, CSS — often 5–20MB total
3
.NET runtime boots inside WebAssembly sandbox C# code is now running inside your browser tab — all rendering & logic is local
4
User clicks / interacts — NO server call for UI Counter increments, dropdowns open, tabs switch — all instant, no network needed for UI logic
5
When data is needed → HTTP call to Web API Uses HttpClient to call /api/products etc. You'll see these as GET/POST requests in the network tab
6
API returns JSON → browser renders update The WASM runtime processes the JSON and re-renders the relevant component client-side

04 — Code Differences

How Code Looks Different

Both use identical Razor component syntax. The difference is in how data is accessed.

🔵 BLAZOR SERVER — Direct DB Access
FetchData.razor razor 
// NO API needed!
@inject MyDbContext db

<h3>Products</h3>

@code {
  List<Product> products;

  protected override async
  Task OnInitializedAsync() {
    // Direct EF Core query!
    products = await db
      .Products
      .ToListAsync();
  }
}
🟡 BLAZOR WASM — Must Use HTTP API
FetchData.razor razor 
// Must go through API
@inject HttpClient Http

<h3>Products</h3>

@code {
  List<Product> products;

  protected override async
  Task OnInitializedAsync() {
    // HTTP call to your API
    products = await Http
      .GetFromJsonAsync<
      List<Product>>(
        "/api/products");
  }
}
⚡ .NET 8+ — Per-Component Render Mode
MyComponent.razor razor 
// Mix modes in the same app!
@page "/admin"
@rendermode InteractiveServer     // secure, server-side

@page "/catalog"
@rendermode InteractiveWebAssembly // fast, client-side

@page "/about"
@rendermode Static                // no interactivity needed

@page "/dashboard"
@rendermode InteractiveAuto        // Server first, then WASM
05 — Feature Comparison

Full In-Depth Comparison

Factor 🔵 Blazor Server 🟡 Blazor WebAssembly
Initial load speed ✓ Very fast — only small JS file downloaded ✗ Slow first load — 5–20MB download (dotnet.wasm + DLLs)
Subsequent interactions ✗ Network round-trip per interaction via WebSocket ✓ Instant — runs locally, no server call for UI
Offline support ✗ None — if WebSocket drops, app freezes ✓ Full offline possible (PWA) after first load

Secrets / API keys ✓ Safe — code never leaves the server ✗ Exposed — DLLs can be decompiled by users
Scalability ✗ Challenging — 1 WebSocket (circuit) per user = memory per user ✓ Easier to scale — stateless API, can use CDN + load balancing
Latency Depends on network quality (round-trip to server) Depends on device speed (processing power)
Browser support ✓ All browsers including IE11 — only needs JS Modern browsers only — requires WebAssembly support
Mobile apps / multi-client Poor fit — tied to browser WebSocket ✓ Perfect — shared API used by mobile, web, etc.
Debugging ✓ Easy — standard .NET server debugging Harder — needs special browser debug proxy setup
Real-time updates ✓ Built-in — WebSocket already open Requires extra SignalR setup from client to server
SEO (search crawlers) ✓ Better — server renders HTML directly Poor by default — needs prerendering workaround

Desktop app migration ✓ Great fit — direct service/DB access like WinForms More refactoring needed
Streaming rendering ✓ Supported — show UI shell while data fetches Not applicable — app loads fully first

06 — .NET 8+ Render Modes

Modern Blazor Render Modes

From .NET 8, Blazor introduced per-component render modes — you're no longer forced to pick one model for the whole app.

Static SSR
@rendermode Static
No interactivity. Pure HTML from server. Great for read-only pages, SEO-critical content. Zero JS overhead.
InteractiveServer
@rendermode InteractiveServer
Component runs on server via SignalR. Fast startup, direct DB access. Best for admin panels, forms.
InteractiveWebAssembly
@rendermode InteractiveWebAssembly
Component downloads and runs in browser. Offline capable after load. Best for interactive, data-heavy UI.
InteractiveAuto
@rendermode InteractiveAuto
Best of both: starts as Server for fast initial load, then transparently switches to WASM once downloaded. Premium UX.
07 — When to Use Which

Real-World Use Cases

🔵
USE BLAZOR SERVER WHEN...
  • Building internal LOB (Line-of-Business) apps for employees on stable corporate networks
  • Migrating a desktop app (WinForms/WPF) to web — direct DB access is familiar
  • Building admin dashboards with sensitive business logic that must stay on server
  • Need real-time multi-user collaboration (chat support, monitoring dashboards)
  • Targeting older browsers or low-powered kiosk devices that can't run WASM
  • Building a quick MVP or prototype where simplicity of a single project matters
  • Financial apps with strict security — code is never sent to the client
  • Apps with heavy EF Core database operations — no API boilerplate needed
  • Need SSR for SEO on a CMS-like content app
🟡
USE BLAZOR WEBASSEMBLY WHEN...
  • Building a public-facing SPA where scalability matters (many concurrent users)
  • You plan to support mobile apps later — shared API serves both web and native
  • Need offline / PWA support (field workers, remote areas with spotty connectivity)
  • Building tools/apps that are CPU-intensive on the UI — canvas, games, local processing
  • Want to host on cheap static CDN (Netlify, Azure Static) without a persistent server
  • Already have or are building a REST API — WASM is a natural frontend for it
  • Want multiple frontend clients (React, Flutter, WASM) sharing one API backend
  • Building a developer tool or design tool (Figma-like) where client-side speed matters
  • e-Commerce storefront where browsing speed post-load is critical

09 — Minor Details & Gotchas

Tips, Traps & Small Details

🔵 Blazor Server — Circuit & Memory Each connected user holds a "circuit" in server RAM. 1000 users = 1000 persistent WebSocket connections + 1000 in-memory state objects. Plan server RAM accordingly.
🟡 WASM — Initial Load is Cached The heavy 5–20MB download happens only once. After that, browsers cache dotnet.wasm and DLLs aggressively. Subsequent visits feel much faster.
⚠️ Blazor Server — Connection Drop If the WebSocket connection drops (bad wifi, server restart), the UI shows a "Attempting to reconnect..." spinner. The app is completely frozen until reconnection. Design for this gracefully.
🟡 WASM — No Secrets in Code Never put API keys, DB connection strings, or sensitive logic in WASM. Users can download and decompile your DLLs using tools like ILSpy. Always validate on the server/API.
🔵 Blazor Server — Scaling Out is Hard Adding multiple server instances behind a load balancer requires sticky sessions (a user must always hit the same server) or using Azure SignalR Service as a backplane.
⚡ .NET 8 Auto Mode — Best of Both InteractiveAuto starts with Server for fast first load, then silently downloads WASM in the background. Next navigation switches to WASM. Transparent to the user, best UX.
🔵 Streaming Rendering Blazor Server supports Streaming Rendering — send the HTML shell immediately, then stream in the data parts as they resolve. Perceived performance is dramatically better for slow data pages.
🟡 WASM AOT Compilation Ahead-Of-Time (AOT) compilation compiles C# directly to native WASM bytecode. Massive runtime performance gain, but increases download size further. Enable with RunAOTCompilation=true.
⚡ Shared Code Between Models The biggest Blazor superpower: same Razor component files work with any hosting model. Business logic in a shared library is reusable across Server, WASM, and Blazor Hybrid (MAUI).
⚠️ WASM NuGet Compatibility Not all NuGet packages work in WASM because the browser sandbox blocks many OS-level APIs (file system, sockets, etc.). Always check a package for "browser compatible" or "blazorwasm" tags before using.
10 — TL;DR

Quick Summary

🔵 Blazor Server in One Sentence

Your C# code runs on the server, and the browser is just a dumb terminal that receives binary DOM patches over a permanent WebSocket — fast to start, powerful, but requires a constant server connection.

🟡 Blazor WASM in One Sentence

Your entire .NET app downloads and runs inside the browser — slow to start but blazingly fast after, offline-capable, scales cheaply, but every server operation needs a proper API endpoint.

THE GOLDEN RULE

"Start with Blazor Server for simplicity. Choose WebAssembly when you need offline support, plan to build mobile clients, or need to scale to thousands of users with cheap hosting. Use InteractiveAuto in .NET 8+ when you want the best of both worlds."
