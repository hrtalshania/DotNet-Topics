01

What is Dependency Injection?

💡 Core Idea

Dependency Injection is a design pattern where a class does not create its own dependencies. Instead, dependencies are provided from outside — usually via the constructor.

Think of it like a restaurant: the chef (your class) doesn't go to the farm to get ingredients. Someone else brings them. The chef just cooks.

❌ Without DI — Bad Practice
class OrderService
{
  private CardPaymentService _payment
    = new CardPaymentService();
  // Tight coupling!
}

Problems: hard to test, hard to replace, tightly coupled.

✅ With DI — Correct
class OrderService
{
  private readonly IPaymentService _payment;

  public OrderService(
    IPaymentService payment)
  {
    _payment = payment;
  }
}

Benefits: loose coupling, testable, swappable.

✅ Benefits of DI
  • 🔗
    Loose coupling — classes don't depend on concrete implementations
  • 🧪
    Easy testing — swap real services with fakes/mocks
  • 🔄
    Replace behavior without touching business logic
  • 🛠️
    Maintainability — clear separation of concerns
02

Why Interfaces Are Mandatory

Interfaces are the backbone of DI. They define a contract that your classes depend on — not a concrete implementation. This is what makes DI powerful.

C# — Interface Definition
public interface IPaymentService
{
    void Pay(decimal amount);
}

// Implementation 1 — Real
class CardPaymentService : IPaymentService
{
    public void Pay(decimal amount) { /* real logic */ }
}

// Implementation 2 — Fake (for testing)
class FakePaymentService : IPaymentService
{
    public void Pay(decimal amount) { /* simulated logic */ }
}

// The business layer never knows which one it's using
🎯 What Interfaces Enable
  • 🔀
    Multiple implementations — real, fake, mock
  • ⚡
    Runtime switching — change behavior via config
  • 🧪
    Mocking/faking during unit tests

04

Service Implementations

C# — Logger (Singleton)
class ConsoleLoggerService : ILoggerService
{
    private readonly Guid _id = Guid.NewGuid();
    // _id stays same for entire app lifetime (Singleton)

    public void Log(string message)
    {
        Console.WriteLine($"[Logger {_id}] {message}");
    }
}
C# — Payment (Transient) & Fake
class CardPaymentService : IPaymentService
{
    private readonly Guid _id = Guid.NewGuid();
    // _id changes each time DI creates a new instance (Transient)

    public void Pay(decimal amount)
    {
        Console.WriteLine($"[Payment {_id}] Paid {amount} via Card");
    }
}

// Fake version — for testing only
class FakePaymentService : IPaymentService
{
    private readonly Guid _id = Guid.NewGuid();

    public void Pay(decimal amount)
    {
        Console.WriteLine($"[FAKE PAYMENT {_id}] Simulating {amount}");
    }
}
C# — Repository (Scoped)
class OrderRepository : IOrderRepository
{
    private readonly Guid _id = Guid.NewGuid();
    // _id stays same within one request scope (Scoped)

    public void Save(Order order)
    {
        Console.WriteLine($"[Repo {_id}] Order saved");
    }
}
C# — OrderService (Business Layer)
class OrderService : IOrderService
{
    private readonly IPaymentService _paymentService;
    private readonly ILoggerService _loggerService;
    private readonly IOrderRepository _orderRepository;

    public OrderService(
        IPaymentService paymentService,
        ILoggerService loggerService,
        IOrderRepository orderRepository)
    {
        _paymentService = paymentService;
        _loggerService = loggerService;
        _orderRepository = orderRepository;
    }

    public void ProcessOrder(Order order)
    {
        _loggerService.Log("Order processing started");

        _paymentService.Pay(order.Amount);
        _paymentService.Pay(300);       // Same instance — NOT a new object

        _orderRepository.Save(order);

        _loggerService.Log("Order processing completed");
    }
}
05

Manual DI — The Old Way

Before DI containers existed, developers wired everything by hand. This works, but it becomes painful fast.

C# — Manual Wiring
var logger  = new ConsoleLoggerService();
var payment = new CardPaymentService();
var repo    = new OrderRepository();

var orderService = new OrderService(payment, logger, repo);

var order = new Order { Amount = 500 };
orderService.ProcessOrder(order);
Problems: Object creation is scattered everywhere, hard to switch implementations, and poor testability. As the app grows, this becomes unmanageable.
06

DI Container — The Correct Way

The DI container (ServiceCollection in .NET) handles all object creation for you. You just register your services once and let the container resolve them.

C# — Service Registration
var services = new ServiceCollection();

// Register with lifetimes
services.AddSingleton<ILoggerService, ConsoleLoggerService>();
services.AddTransient<IPaymentService, CardPaymentService>();
services.AddScoped<IOrderRepository, OrderRepository>();
services.AddScoped<IOrderService, OrderService>();

var serviceProvider = services.BuildServiceProvider();
C# — Resolving Services (Per Request / Scope)
for (int i = 1; i <= 2; i++)
{
    Console.WriteLine($"\n--- REQUEST {i} ---");

    using var scope = serviceProvider.CreateScope();

    var orderService = scope.ServiceProvider
                            .GetRequiredService<IOrderService>();

    var order = new Order { Amount = 500 };
    orderService.ProcessOrder(order);
}
// Scoped services are disposed at end of each using block
Key insight: Always resolve scoped services inside a CreateScope() block. Never resolve scoped services directly from the root provider — this causes memory leaks.
07

Service Lifetimes

Singleton
AddSingleton
One instance shared across the entire application lifetime. Created once, reused everywhere.
✅ Logger, Configuration, Cache
Scoped
AddScoped
One instance per request or scope. Same object within a request, new object for next request.
✅ Repository, DbContext, UnitOfWork
Transient
AddTransient
New instance every time the service is resolved from the DI container.
✅ Payment service, Email sender
Tip: The GUID trick in the code examples makes lifetimes observable — the same GUID = same instance. Different GUID = new instance.
08

Scoped vs Transient — Key Confusion

This is the most common confusion in DI. Both create new instances across requests — but their behavior within a request is very different.

⚡ TRANSIENT — New per resolution
// Two different instances in same request
OrderService     → PaymentService (A)
InventoryService → PaymentService (B)

Each service that needs IPaymentService gets its own fresh instance.

🔵 SCOPED — Same per request
// Same instance shared within one request
OrderService     → Repository (X)
InventoryService → Repository (X)

All services within one scope share the same Repository instance.

🔍 Why PaymentService GUID Is Same When Called Twice
// Inside ProcessOrder:
_paymentService.Pay(500);   // ← NOT a new object
_paymentService.Pay(300);   // ← Same instance, same GUID

// DI injects once at construction time
// Calling a method does NOT trigger DI again

DI happens only at object construction time. Calling methods on an already-injected service does not create new instances — regardless of Transient/Scoped/Singleton.

09

Factory Pattern — New Instance Per Call

If you genuinely need a new instance on every method call (not just per DI resolution), use a factory instead.

C# — Factory Injection
class OrderService : IOrderService
{
    private readonly Func<IPaymentService> _paymentFactory;

    public OrderService(Func<IPaymentService> paymentFactory, ...)
    {
        _paymentFactory = paymentFactory;
    }

    public void ProcessOrder(Order order)
    {
        var p1 = _paymentFactory();   // New instance
        p1.Pay(500);

        var p2 = _paymentFactory();   // Another new instance
        p2.Pay(300);
        // p1 and p2 will have DIFFERENT GUIDs
    }
}
C# — Register Factory
// Register the factory — .NET supports Func<T> automatically
services.AddTransient<IPaymentService, CardPaymentService>();
services.AddTransient<IOrderService, OrderService>();
// The container will automatically resolve Func<IPaymentService>
When to use factories: Use this pattern when you need a fresh object per method call — e.g., a fresh HTTP client per request, a new transaction per operation.
10

Testing with Fake Services

One of the biggest DI benefits: switch your entire payment system from real to fake with one line change — zero changes to business code.

C# — Swap Real → Fake
// Production — real payments
// services.AddTransient<IPaymentService, CardPaymentService>();

// Testing — fake/simulated payments
services.AddTransient<IPaymentService, FakePaymentService>();

// Everything else stays identical ✅
// OrderService never changes ✅
// Business logic untouched ✅
🧪 What This Means for Testing
  • ✔️
    No changes in OrderService — tests are purely configuration
  • ✔️
    No real payments charged — safe to run in CI/CD
  • ✔️
    Same flow tested — only the implementation is swapped
  • ✔️
    This is the real DI benefit — testability by design
11

Critical DI Rules for Production

  • 🚫
    Never inject concrete classes — always depend on interfaces, never new CardPaymentService() inside business classes
  • 🚫
    Never resolve scoped services from the root provider — always use CreateScope(), otherwise scoped services behave like singletons (memory leaks)
  • 🚫
    Never modify business logic for testing — if you have to change OrderService to test it, your DI is wrong
  • ⚠️
    All registrations before BuildServiceProvider() — you cannot register services after the provider is built
  • ✅
    DbContext must always be Scoped — one DbContext per request, never Singleton (causes concurrency issues)

Mental Model — The Full Picture

Program.cs — Composition Root
  ├─ Register services (AddSingleton / AddScoped / AddTransient)
  ├─ BuildServiceProvider()
  ├─ CreateScope() — one per request
  │   ├─ Resolve IOrderService
  │   │   ├─ DI injects IPaymentService (Transient → new instance)
  │   │   ├─ DI injects ILoggerService (Singleton → shared instance)
  │   │   └─ DI injects IOrderRepository (Scoped → same per request)
  │   ├─ Run ProcessOrder()
  │   └─ Dispose scoped services at end of using
  └─ Repeat for next request
🔑 Summary — Lock These In
  • 1️⃣
    DI = control inversion — the container creates objects, not you
  • 2️⃣
    Interfaces enable runtime switching — swap implementations without touching business code
  • 3️⃣
    Lifetimes control reuse — Singleton (app), Scoped (request), Transient (per resolution)
  • 4️⃣
    Scoped ≠ Transient — Scoped shares within a request, Transient never shares
  • 5️⃣
    DI container replaces manual wiring — no more new in business code
  • 6️⃣
    Testing = just configuration — swap implementations at registration, business code unchanged
